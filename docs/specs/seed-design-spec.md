# Seed Design Spec — single-hop datasets for Figures 5/6

**Status:** spec, ready to implement.
**Companion:** `harness-fix-spec.md` (same directory) — defines how `main()` consumes these
specs (re-seed per param point, eval-user namespacing, post-seed verification).
**Provisional-grounding warning:** items tagged `[CONTINGENT-§1.3]` or `[CONTINGENT-AUDIT]`
depend on SOURCE_OF_TRUTH §1.3 / audit findings that a parallel session may revise.

---

## 1. Goals and invariants

1. **Neutrality.** One backend-agnostic seed spec per sweep point; each adapter translates it
   to its store. Nothing in the data shape favors the graph model: the measured query is
   single-hop own-tweets, where SQL and Neo4j are expected to **tie** Jac. (The graph model's
   claimed win is multi-hop — Phase 4.5, out of scope here.) No artificial channel noise; the
   original benchmark's single-hop Channel-edge noise (SOURCE_OF_TRUTH §4,
   `no-othernode-noise`) is deliberately not reproduced.
2. **Determinism.** Same generator version + same point → byte-identical spec file. All four
   backends seeded from the same file hold content-identical datasets, enabling Phase-5
   cross-system result comparison.
3. **Exact selectivity.** The fraction of tweets matching the predicate is exact by
   construction (counted, not sampled) — zero sampling noise on the x-axis.
4. **Sweep realized by data only.** The timed request is `POST /walker/load_own_tweets` with
   `{}` on every backend at every point (harness-fix-spec §2). Fan-out and selectivity exist
   only in the seeded data.

## 2. The predicate: a true field predicate, `like_count > K`

The selectivity axis needs a **field predicate** evaluated inside each storage engine — that is
what Filter Pushdown means (FP evaluates field predicates in the storage engine; GTI gives O(1)
typed-neighbor lookup). Today no backend has a filterable field:

| Backend | Likes storage today | "likes > k" today would be |
|---|---|---|
| Jac | `Like` edges Profile→Tweet (`server.jac:54-58`), no count field | edge traversal + Python-side `len()` — not a field predicate, FP cannot push it |
| Postgres | `likes` join table (`schema.sql:73`) | `HAVING COUNT(*)` over join rows — aggregate, not field |
| SQLAlchemy | `like_table` (`models.py:27`) | same, or Python-side after `selectinload` |
| Neo4j | `t.likes` string array (`main.py:113-124`) | `size(t.likes) > k` — array scan |

Three different cost models = a non-neutral axis. **Therefore: add a denormalized
`like_count: int` field to Tweet on all four backends**, and the predicate is the scalar
comparison `like_count > K`:

- Jac: `has like_count: int = 0;` on `node Tweet` (`server.jac:39`) — FP can push a scalar
  field predicate; this is the mechanism the paper claims.
- Postgres: `like_count BIGINT NOT NULL DEFAULT 0` column on `tweets` (`schema.sql:50`);
  predicate in SQL `WHERE`.
- SQLAlchemy: `like_count: Mapped[int]` on `Tweet` (`models.py:99`); predicate in the
  `select().where()`.
- Neo4j: `like_count` integer property on Tweet; predicate in Cypher `WHERE`.

**Constants:** `LIKES_THRESHOLD K = 10`. `SPEC_VERSION = 1`. Matching tweets carry
`like_count` uniform in `[11, 20]`; non-matching uniform in `[0, 10]` (10 > 10 is false, so
the bands are airtight). Max likes per tweet = 20 → liker pool of 32 distinct users suffices
(PG's `(tweet_id, user_id)` PK requires distinct likers per tweet, not across tweets).

The likers list (usernames) is still seeded in full — `len(likers) == like_count` always —
because the response payload includes liker names (payload realism) and Phase 5 cross-checks
`like_count == len(likes)` as a data-integrity probe.

**`load_own_tweets` must apply the predicate server-side on all four backends** (this is the
server-change keystone — see §8). The walker/route stays parameterless; `K` is a server-side
constant. Idiomatic implementations:

- Jac: filtered edge ref with field predicate over typed neighbors —
  `[my_profile-->(?:Tweet)](?like_count > 10)` (exact syntax per the filter_pushdown build;
  the point is: predicate in the traversal expression so FP/GTI can act on it, NOT a Python
  `if` inside the loop). `[CONTINGENT-AUDIT]` — C13 ("FP actually reduces work") is confirmed
  today; if the parallel audit revises it, expectations change but this design does not.
- Postgres: `... WHERE t.author_id = %s AND t.like_count > 10 ORDER BY t.created_at DESC`.
- SQLAlchemy: `.where(Tweet.author_id == uid, Tweet.like_count > 10)`.
- Neo4j: `MATCH (p:Profile {jac_id: $uid})-[:POST]->(t:Tweet) WHERE t.like_count > 10 ...`.

## 3. Neutral seed spec — JSON schema

One file per sweep point: `seed/{sweep_type}_{param_value}.json`, committed to the repo
(deterministic, reviewable, ~0.5–1.5 MB each). Generated by `DBaseRunner/seed_gen.py` (new
file, §4).

```json
{
  "spec_version": 1,
  "sweep_type": "fanout",            // "fanout" | "selectivity"
  "param_value": 250,
  "rng_seed_string": "v2:fanout:250",// the string hashed to seed the RNG (provenance)
  "likes_threshold": 10,
  "n_tweets": 250,
  "selectivity_pct": 25,             // fanout sweep fixes 25 (SPEC_VERSION 2); selectivity sweep = param_value
  "expected_matching": 63,           // EXACT count of tweets with like_count > 10
  "expected_matching_keys": ["t_0007", "t_0019", ...],   // len == expected_matching
  "likers": ["liker_000", "...", "liker_031"],           // pool, run-agnostic names
  "eval_user_suffix": "fanout_250",  // harness prefixes bench_{run_id}_ at seed time
  "tweets": [
    {
      "key": "t_0000",                       // stable cross-backend identity
      "content": "[t_0000] <deterministic text, ~100-140 chars>",
      "created_at": "2026-01-01T00:00:00Z",  // base + index minutes, strictly monotonic
      "like_count": 14,
      "likers": ["liker_003", "liker_007", "..."],   // len == like_count, sampled from pool
      "comments": [
        {"author": "liker_002",
         "content": "<deterministic, 40-80 chars>",
         "created_at": "2026-01-01T00:00:30Z"}        // tweet.created_at + 30s*(j+1)
      ]
    }
  ]
}
```

Schema rules:

- `key` is the cross-backend identity. Backend-native tweet IDs diverge in type and value
  (int vs jac id vs generated string — audit obs 480), so the key is **embedded as a content
  prefix** `[t_NNNN] ` and Phase 5 matches on it. No backend needs to store the key as a
  separate field.
- `created_at` values are explicit and synthetic (base `2026-01-01T00:00:00Z` + 60 s × tweet
  index). Servers MUST store the provided value, not `now()` (PG/SQLA columns currently
  default `now()` — import must override; §8).
- Liker names are **run-agnostic** (`liker_000`…`liker_031`) and shared across all points;
  import is upsert-by-username (§8), so re-runs and multiple points never conflict. Eval
  users are run-namespaced instead (harness-fix-spec §1.2); that alone prevents dataset
  collisions, since all seeded tweets hang off the eval user.
- `comments[].author` is the canonical comment-author key; adapters map it to each server's
  field (`author_handle` in PG, `handle` in SQLA, `username` in jac's embedded dicts).

## 4. Generator algorithm (`seed_gen.py`)

CLI: `python seed_gen.py --out seed/` → writes all 10 point files + `seed/manifest.json`
(list of files, spec_version, K, per-point `expected_matching`). No flags vary the data;
changing the data means bumping `SPEC_VERSION` in the source.

Per point `(sweep_type, param_value)`:

1. **RNG.** `rng = random.Random(f"v{SPEC_VERSION}:{sweep_type}:{param_value}")` — Python
   seeds `Random` from the string hash deterministically across runs/machines
   (`PYTHONHASHSEED` does not affect `random.Random(str)` seeding — it uses the value, not
   `hash()`; engineer: verify on the pinned Python (3.12) once, or seed from
   `int.from_bytes(hashlib.sha256(s.encode()).digest()[:8], "big")` to be immune; the sha256
   route is RECOMMENDED and assumed below).
2. **Dimensions.**
   - fanout sweep: `n_tweets = param_value`, `selectivity_pct = 25` (raised from 5% in SPEC_VERSION 2 per HARNESS_REVIEW L2 — clears the ≥20-matching floor at the lowest point).
   - selectivity sweep: `n_tweets = 1000`, `selectivity_pct = param_value`.
   - `n_matching = (n_tweets * selectivity_pct + 50) // 100`  (integer round-half-up; exact
     table in §5).
3. **Matching assignment.** `matching_idx = set(rng.sample(range(n_tweets), n_matching))` —
   matching tweets scattered through the timeline, not clustered (avoids any
   ordered-storage backend getting a contiguous-range advantage).
4. **Per tweet `i` (in index order — one RNG stream, fixed draw order; never reorder draws
   without bumping SPEC_VERSION):**
   - `like_count = rng.randint(11, 20)` if `i in matching_idx` else `rng.randint(0, 10)`.
   - `likers = sorted(rng.sample(LIKER_POOL, like_count))` (sorted → list order itself is
     never a cross-backend comparison hazard).
   - `n_comments = rng.randint(0, 3)`; each comment: author `rng.choice(LIKER_POOL)`,
     content from the word list, `created_at = tweet_ts + 30s*(j+1)`.
   - `content = f"[t_{i:04d}] " + words(rng, target≈100–140 chars)` from a fixed word list
     embedded in `seed_gen.py` (~200 common words; no external deps, no locale dependence).
5. **Emit** JSON with `sort_keys=True, indent=1` (stable bytes → meaningful git diffs), plus
   the manifest.

Generator self-checks (assert before writing): `len(expected_matching_keys) ==
expected_matching`; every matching tweet `like_count > K`; every non-matching `<= K`; all
`likers` lists duplicate-free; byte-determinism test in CI (generate twice, compare digests).

## 5. Sweep tables (normative)

Fan-out sweep (Fig 5) — selectivity fixed at 25% (SPEC_VERSION 2):

| param_value (n_tweets) | 100 | 250 | 500 | 750 | 1000 |
|---|---|---|---|---|---|
| expected_matching | 25 | 63 | 125 | 188 | 250 |

Selectivity sweep (Fig 6) — n_tweets fixed at 1000:

| param_value (%) | 10 | 25 | 50 | 75 | 100 |
|---|---|---|---|---|---|
| expected_matching | 100 | 250 | 500 | 750 | 1000 |

Totals per backend per run: 2,600 (fanout) + 5,000 (selectivity) = 7,600 tweets;
~15.5 likes/tweet avg on matching, ~5 on non-matching; ~1.5 comments/tweet. Largest response
(selectivity=100%): 1000 tweets × ~0.5 KB ≈ 0.5 MB — `response_bytes` in the CSV becomes a
direct selectivity diagnostic (should scale ~linearly with `expected_matching`; a flat
`response_bytes` across points is the no-op-sweep failure mode resurfacing — plot it in Phase 4
sanity checks).

Axis definition (normative, resolves the wording conflict flagged in harness-fix-spec §6.1):
**fan-out = number of tweets authored by the eval user = out-degree of the user's own `POST`
edges.** Not followees. HARNESS_REBUILD.html's "followees" wording is stale.

## 6. Determinism & Phase-5 hooks

- Same spec file → all four backends hold the same logical dataset: identical content strings
  (keys embedded), `created_at`, `like_count`, liker-name lists, comments.
- Phase 5 comparison contract enabled here: join on the `[t_NNNN]` content key; compare
  content, `created_at`, `like_count`, sorted likers, comment multiset. Compare
  **order-insensitively**: PG/SQLA order by `created_at DESC` today; jac and neo4j do not
  order at all. Decision: do NOT add ORDER BY to jac/neo4j for the timed path — it would add
  unequal sort costs (Python sort in-walker vs index-backed DB sort) to a latency benchmark
  mid-rebuild. Phase 5 sorts client-side by key. `[CONTINGENT-AUDIT]` — if re-validation
  decides ordering belongs in the measured work, add `ORDER BY created_at DESC` to all four
  simultaneously and re-baseline; never to a subset.

## 7. Re-seed / reset flow between param points

Defined normatively in harness-fix-spec §1; summary of the data-side view:

1. Fresh eval user per point (`bench_{run_id}_{sweep}_{param}`) — no deletion needed anywhere;
   the only backend without a delete endpoint (jac) is thereby fully supported.
2. Likers are global, upserted on first reference, shared by all points.
3. Double-seed guard + exact post-seed verification (count, threshold, key set) make every
   seeding self-checking; a wrong/partial import cannot silently produce a timed point.
4. `clear_data` (PG `db.py:121`, SQLA `__init__.py:135`, neo4j `main.py:190`) is only used by
   the optional `--reset` at run start; it is hygiene, not correctness.

## 8. Per-backend server endpoint requirements

The seed path is: `ensure_user` → `setup_profile` → `import_data` (one bulk call per point,
the translated spec as body). `like_tweet`/`add_comment` endpoints are NOT used for bulk
seeding (35k+ individual HTTP calls would be absurd) but must stay consistent with the new
field (R6). Requirements; "NEW" = does not exist today:

**All four backends**
- R1. `like_count` field on Tweet (NEW everywhere) — §2.
- R2. `load_own_tweets` applies `like_count > 10` server-side, idiomatically (NEW everywhere) — §2.
- R3. `import_data` accepts the translated point spec: tweets with `content`, `created_at`
  (stored verbatim, not `now()`), `like_count`, `likers[]`, `comments[]`; plus the liker pool.
  Single transaction / single walker execution per call where the store supports it.
- R4. `import_data` upserts likers by username (idempotent); never duplicates an existing one.
- R5. `import_data` attaches all tweets to the **eval user** (see per-backend notes).
- R6. `like_tweet` additionally increments `like_count` (and `add_comment` keeps whatever
  invariant the backend's comment storage has) so the denormalized field can never silently
  drift from ground truth in later phases.

**Jac (`servers/jac/server.jac`)** — `import_data` exists (`server.jac:946`, `:pub`) with an
ad-hoc `data` schema; engineer must read and extend it:
- Called WITH the eval user's JWT; tweets created under the **caller's** Profile via `Post`
  edges. Depends on `:pub`+token root binding (harness-fix-spec §6.3); if `:pub` won't bind,
  make `import_data` auth-required.
- Likers: create detached `Profile` nodes (username only, upsert) — NOT full registered users
  (no jac-cloud accounts needed); `Like` edges liker-Profile → Tweet. `load_own_tweets`'s
  liker read `[tweet<-:Like:<-(?:Profile)]` (`server.jac:894`) works unchanged.
- Set `tweet.like_count` from the spec; store `comments` into the existing embedded list field
  (`server.jac:39-46`) with `username` ← spec `author`.
- Volume note: selectivity=100% point ⇒ ~15,500 `Like` edges in one walker call; untimed, but
  confirm it completes within jac-cloud's request timeout on clarity — else chunk the call
  (adapter may split tweets into batches of 250; spec allows multiple `import_data` calls per
  point, R4 makes that safe).

**Postgres (`servers/postgres/`)** — `import_data` exists (`routes/walker.py:528`, unauth):
- Body gains `author_username`; server resolves/creates the eval user row, inserts tweets
  with explicit `created_at` and `like_count`, liker user rows (upsert on `username`),
  `likes` rows, `comments` rows (`author_handle` ← spec `author`). One transaction.
- Schema: `ALTER TABLE tweets ADD COLUMN like_count BIGINT NOT NULL DEFAULT 0` + R2 predicate
  in the `load_own_tweets` SQL (`routes/walker.py:504`).
- Index policy (hand-tuned-PG status): RECOMMENDED extend the covering index to
  `(author_id, created_at DESC) INCLUDE (content, like_count)` (`schema.sql:61`) so the
  predicate evaluates inside the index-only scan. Whatever is chosen, record it in the run
  metadata sidecar — it is part of the "hand-tuned" claim. OPEN QUESTION for the audit owner:
  is extending the index in-scope for "hand-tuned PG," or must PG run the predicate from the
  heap? Default: extend.
- **R7 (PG only): add `GET /health`** — `backends/postgres.py:30` calls it; the server has no
  health route today (SQLA has one at `__init__.py:115`; copy the pattern).

**SQLAlchemy (`servers/sqlalchemy/`)** — `import_data` exists (`walker.py:336`, unauth):
- Same as Postgres via ORM: `Tweet.like_count` mapped column, bulk inserts
  (`session.add_all` / `bulk_insert_mappings`), liker upserts, `Comment.handle` ← spec
  `author`, explicit `created_at`. R2 predicate in `walker.py:313-320` query.

**Neo4j (`servers/neo4j/src/main.py`)** — `import_data` exists (`main.py:298`, unauth):
- Tweets created with `POST` edges from the eval `Profile` (`jac_id` = eval username — the
  username IS the id in this server), properties `content`, `created_at`, `like_count`,
  `likes` = liker-name array, `comments` = existing JSON-string encoding.
- Likers need no nodes (likes are an array property — keep it; it is the idiomatic model for
  this server and the response payload only needs names). Phase 5 sees identical liker name
  lists either way.
- R2 predicate in the Cypher (`main.py:153-159`): `WHERE t.like_count > 10`.
- Use `UNWIND $tweets AS t CREATE ...` batch form, not per-tweet round-trips.

**Adapter `seed(spec)` translation summary** (`backends/*.py`, harness-fix-spec §3): each
adapter reshapes the neutral spec into its server's `import_data` body, prefixing nothing —
all naming (eval username, liker names) is already resolved by harness + spec. Jac adapter
must send the call on the authenticated session; PG/SQLA/neo4j import endpoints are unauth and
take `author_username` explicitly.

## 9. Failure modes this design closes (traceability)

| Original audit finding | How this seed design answers it |
|---|---|
| sweep-is-data-no-op (every point identical data) | dataset IS the sweep point; verified counts per point (§5, harness §1.4) |
| selectivity-axis-mislabeled-for-baselines | one axis definition, same predicate constant, same data, all backends (§2, §5) |
| no-othernode-noise / channel-noise asymmetry | no noise edges at all; fan-out = own POST edges only (§1, §5) |
| payload-build asymmetry | identical likes/comments content seeded everywhere; payload realism uniform (§3) |
| empty-second-hop | not applicable by construction — single-hop is the declared scope; multi-hop deferred to Phase 4.5 (§10) |
| ID-type divergence blocking cross-checks (obs 480) | content-embedded `[t_NNNN]` keys (§3, §6) |

## 10. Deferred: Phase 4.5 hooks

`hop_depth` is excluded from this spec's generator output. Reserve, do not implement:
`sweep_type: "hop_depth"` spec files would add a `follows` section (eval user → N followees,
each authoring tweets) and target `load_extended_feed`. The generator's point loop and the
spec schema are shaped so this is an additive change (new `sweep_type` branch + new optional
top-level key), not a format break. Bump `spec_version` when it lands.
