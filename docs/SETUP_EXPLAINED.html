Why any of this exists

Running a server boils down to one sentence: "I have some code; I need a computer somewhere to run it, with the right software installed, reachable over the network — and I want it to stay up." Every confusing word below (Docker, pod, minikube…) is just a tool that solves one slice of that sentence. Read this once and the rest of the JacDB docs stop looking like alphabet soup.

Our goal: run four backends (jac, postgres, sqlalchemy, neo4j) on one machine called clarity, point one benchmark harness at each, and measure how fast each answers the same question. Everything here is the plumbing that gets those four servers alive and reachable.

The 8 words you actually need

In dependency order — each builds on the one above. The italic line is a plain-life analogy.

📦
Docker image — a frozen template
A snapshot of a whole computer's hard drive: the operating system, the installed software (Python, Postgres…), and your code, all baked into one file. Nothing is running — it's just a saved template you can boot later.
Like a saved disk image, or a cake recipe's finished photo — the thing you copy from.
▶️
Docker container — a running copy of an image
You take the frozen image and "boot it up." It's now live and serving requests. One image can spawn many containers.
The recipe (image) vs. the actual cake you baked from it (container). You can bake many cakes from one recipe.
📝
Dockerfile — the recipe that builds an image
A short text file: "start from Python 3.12, copy my code in, install these packages." Running docker build follows the recipe and produces an image.
The written recipe steps. docker build = actually cooking it into the finished template.
🚢
Kubernetes — "k8s", the manager that runs containers for you
You don't start containers by hand. You hand Kubernetes a YAML file saying "keep this image running, reachable on this port," and it does it — and restarts it automatically if it crashes.
A restaurant manager: you write the menu (YAML), they staff the kitchen, replace anyone who quits, and keep service running.
🫙
Pod — Kubernetes' word for "one running unit"
Usually one container. "The postgres-app pod" = the container running our Postgres API code. Important: pods are disposable. Kubernetes kills and recreates them freely, and each new one gets a fresh network address.
One cook on shift. If they leave, the manager hires an identical replacement — same job, different person.
🪧
Service — a stable name/address for a pod
Because pods come and go with new addresses, you never talk to a pod directly. You talk to its Service (e.g. postgres-app:8000) and Kubernetes routes you to whichever pod is alive right now.
The restaurant's phone number. It never changes, even when the cook on shift does.
🌐
Cluster — the whole Kubernetes installation
The machines plus the manager, together. "The clarity cluster" = Kubernetes running on the clarity machine.
The entire restaurant building + staff, not one cook.
🧪
minikube — a tiny, single-machine Kubernetes for testing
Real Kubernetes spreads across a datacenter. minikube fakes the entire thing on one computer so you can develop. We run minikube on clarity. It has one quirk that burned us (see Problem A).
A food-truck-sized version of the full restaurant — same idea, one location, for practice.

Our setup: the two-pod shape

Every backend follows the same shape on purpose: an app pod (the API code the harness talks to) and a separate DB pod (the actual database storing data). Keeping them split means all four backends have an identical topology — that fairness is one of the harness's design fixes.

harness (the client / stopwatch) ──▶ app pod (API code) ──▶ DB pod (database)

The harness is the Python program you run that acts as a stopwatch: it sends a request to the app pod, times how long the round-trip takes, and writes a row to a CSV. It is the same code for all four backends — only the backend it points at changes.

Image vs ConfigMap — the difference you noticed

You were right: postgres and neo4j each have their own built Docker image; sqlalchemy does not. The backends differ only in how the app pod's code gets delivered:

BackendApp pod imageHow its code gets inDB pod
postgres custom-built dbaserunner/postgres-app docker build from servers/postgres/Dockerfile stock postgres:15
neo4j custom-built dbaserunner/neo4j-app docker build from servers/neo4j/Dockerfile stock neo4j image
sqlalchemy stock python:3.12-slim (no build) source files mounted via a ConfigMap + pip install at startup stock postgres:16-alpine
jac special — self-deploys see next section MongoDB + Redis (self-deployed)

What is a ConfigMap, and why the difference?

A ConfigMap is just "a bundle of text files Kubernetes hands to a pod." Since sqlalchemy's code is pure Python text, we skip building an image entirely: boot a stock Python pod, drop the .py files in via a ConfigMap, and let a small "init container" run pip install before the app starts. Postgres/neo4j needed a real build (their setup isn't just loose text files), so they get baked images.

ConfigMap upside (sqlalchemy/jac): editing the code = just update the ConfigMap. Fast, no rebuild.
Image cost (postgres/neo4j): editing the code = rebuild the image with a new tag. Slower — and the source of Problem A below.

Why Jac is the odd one out

Jac does not use our k8s/jac/ manifest (that one is broken — see Problem D). Instead servers/jac/deploy.sh runs jac start --scale, and jac-cloud deploys its own entire stack: the app pod, a MongoDB pod (its long-term storage, called "L3"), a Redis pod (a faster cache layer, "L2"), plus ingress and monitoring. You build nothing — Jac does it all. That self-deploying magic is exactly why Jac took the most debugging.

Jac's three storage layers (you'll see L1/L2/L3 everywhere):
  • L1 — an in-memory dictionary inside the app pod itself. A hit here is a local lookup with zero network trip. This is Jac's structural speed edge.
  • L2 — the Redis pod. A hit is one network hop + unpacking the data.
  • L3 — the MongoDB pod. The permanent store; a hit is a hop + a real database query.

What broke on the live session — and the fix for each

One theme runs through almost all of these: stale state and leftover processes. A clean start (next section) prevents most of them.

A · "I rebuilt the image but nothing changed" cost the most time
What happenedminikube is set to imagePullPolicy: Never ("never download, only use what's local"). Combined with the tag :latest, rebuilding under the same tag left the pod serving the old code — Kubernetes saw "I already have :latest" and never noticed the contents changed.
WhyA tag like :latest is just a label. Re-labeling new contents with an existing label doesn't force Kubernetes to reload.
FixNever reuse a tag. Every rebuild gets a fresh number (:v2, :v3…) and the YAML is updated to match. Build directly inside minikube with eval $(minikube docker-env) so the image lands where minikube can see it. Then confirm the pod is on the new tag before trusting any result.
B · curl keeps hitting the wrong server port squatter
What happenedTo reach a pod from the command line we use kubectl port-forward, which opens a local port (say 8000) that tunnels into the cluster. A leftover jac process was already sitting on 8000, so our curl was talking to it, not the pod.
The tellport-forward printed Forwarding from [::1]:8000 (IPv6 only, no 127.0.0.1) — that means the normal address was already taken.
FixForward to a free port (8001), and check first: ss -tlnp | grep :8001 should print nothing. Kill stray jac servers with pkill -9 -f "jac start".
C · Neo4j won't start — "Unrecognized setting DB.SERVICE.PORT" env collision
What happenedKubernetes auto-injects environment variables describing other services into every pod. Because our DB service was named neo4j-db, it set NEO4J_DB_SERVICE_PORT=…. The neo4j image reads any NEO4J_* variable as a config setting, didn't recognize this one, and refused to boot (crash loop).
FixAdd enableServiceLinks: false to the pod spec — "don't inject those env vars." Already committed in the neo4j YAML.
D · Jac's k8s manifest is broken dead end
What happenedOur k8s/jac/ tried the sqlalchemy trick (stock image + inject code), but Jac needs its binary and real installed packages — the init container wiped them out.
FixAbandon that manifest. Use deploy.sh (native jac start --scale) and let jac-cloud deploy itself.
E · Jac prints "Deployment failed: Timeout" — but it's fine false alarm
What happeneddeploy.sh self-checks by curling localhost:30080, which minikube doesn't expose → scary timeout message + it spawns a useless local dev server. The actual pods were healthy the whole time.
FixIgnore the message. Reach Jac via its real service svc/jaseci-service on port 8080. Kill the stray dev servers it spawned (they cause Problem B).
F · Jac API quirks one cycle each
What happenedAuth wants {username, password} (not email); the login token is at body.data.token; tweets come back at body.data.reports[0].tweets; and a walker must be listed in main.jac to be served at all (a missing import gave a 405).
FixAll handled in backends/jac.py + main.jac. Use inspect_schema.py <url> to dump any backend's real auth fields instead of guessing.
G · git push denied auth
What happenedThe laptop's SSH key isn't registered with GitHub, and clarity pushes over HTTPS which needs a Personal Access Token (PAT).
FixCommit locally; Aaron pushes with the PAT. Pull single files with git checkout origin/<branch> -- <path> (grabs just that file, no full merge).
Still open — Jac root isolation. A freshly-registered Jac user currently sees a previous user's data. Root cause is pinned (the benchmark walkers are declared walker:pub → no auth → they all share one hidden system_root). Fixable, not yet fixed. Until it is, Jac numbers aren't trustworthy — but postgres, sqlalchemy, and neo4j are validated and good to run.

The session-start ritual (do this every time)

Pods survive when you close SSH, so you usually don't need to redeploy — just clean up leftovers, check what's alive, re-open a tunnel, and go.

1

Clean the slate

# kill stray jac dev servers (Problem B/E) and old tunnels
pkill -9 -f "jac start"
pkill -f "kubectl port-forward"
# confirm the ports you'll use are free — no output = good
ss -tlnp | grep -E ':800[0-9]'
2

See what's actually running

kubectl get pods   # every line should say Running — not Error / CrashLoop / Pending

If they're all Running, skip straight to step 4. You only redeploy when you changed code.

3

Only if you changed code: redeploy that ONE backend

  • postgres / neo4j (image backends): eval $(minikube docker-env) → docker build -t dbaserunner/<b>-app:vNEXT servers/<b> → bump the tag in k8s/<b>/*.yaml → kubectl apply -f k8s/<b>/. (Problem A — always a new tag.)
  • sqlalchemy / jac (ConfigMap / native): just re-run their deploy script. No build.
  • Always confirm the pod picked up the new tag before trusting results.
4

Open a tunnel to a free port + sanity-check the endpoint

kubectl port-forward --address 127.0.0.1 svc/postgres-app 8001:8000 &
# this catches Problems B + F instantly:
curl -s -o /dev/null -w "%{http_code}\n" -X POST http://127.0.0.1:8001/walker/load_own_tweets

See the cheat sheet below for what each code means.

5

Run with ONE shared run-id, then sanity-check the CSV

uv run python harness.py --backend postgres --url http://localhost:8001 --run-id run01 --sweep fanout selectivity

Use the same --run-id for every backend so their seeded data lines up. Then open the result CSV: response_bytes must grow with fan-out. A flat or zero column means the seed didn't land — stop and investigate before trusting the latencies.

Status-code cheat sheet

When you curl a walker endpoint in step 4, the HTTP status tells you instantly whether you're wired up right:

CodeMeansVerdict
401Right app, route exists, just needs login (the harness logs in for you)GOOD — proceed
200Worked outrightGOOD
405Wrong app on this port (usually a stray jac squatter — Problem B)find it: ss -tlnp | grep :8001
404Route missing — the running image doesn't have this walker (stale build — Problem A)rebuild / re-check tag
000The tunnel isn't connecting at allre-open port-forward
