Key Decisions

TIMING
Updated: both — client-side perf_counter from an in-cluster pod → Service (NOT port-forward) as the identical boundary, PLUS server-side per-phase markers (auth/query/fetch/build). network = client − server. Primary substrate metric = server_total; report client + network too. See HARNESS_CONTEXT.md §5.

All warm. Run 20 warmup requests per config before timing. No clear_cache between trials. Redis L2 is Jac's architecture — measure it as designed.

Timing protocol

import time, requests

def timed_call(session, url, payload):
    t0 = time.perf_counter()
    resp = session.post(url, json=payload)
    t1 = time.perf_counter()
    resp.raise_for_status()
    return (t1 - t0) * 1000  # ms

Phase 4.5 — Multi-hop Benchmark NEW

4.5

Test where graph model actually differentiates

sequential · needs Phase 4 complete

Single-hop results may be competitive. Multi-hop is where Jac's graph traversal vs SQL recursive CTEs vs Neo4j native graph diverges most.

  • Define new endpoint: POST /load_extended_feed — tweets from friends-of-friends (2-hop)
  • Implement on all 4 backends:
    • Jac: natural walker [-->:Follow-->:Follow-->:Tweet]
    • SQL: recursive CTE or self-join
    • Neo4j: native graph traversal (strong here too)
  • Add hop depth as sweep parameter: [1, 2, 3]
  • Add to harness and orchestrate.sh
This makes the paper's graph-model argument empirically grounded — not just "Jac is fast" but "Jac's graph abstraction scales better with traversal depth."

Phase 5 — Correctness Validation

5

Cross-system result comparison

sequential · needs Phase 4 complete

Original correctness check was tautological (each backend checked against its own insert count) and never gated execution. This phase replaces it.

  • After each config point, run the same query on all 4 backends
  • Compare result sets: tweet IDs, counts, ordering
  • Hard fail with detailed log if backends diverge — do not silently continue
  • Check per-tweet likes/comments cardinality (original ignored this)
  • Log correctness results to results/correctness.csv
If backends return different result sets, the latency comparison is meaningless — they're not answering the same question.
